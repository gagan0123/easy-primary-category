# Agents Guide — Easy Primary Category

## Project Overview

A WordPress plugin that allows users to designate a primary (main) category for posts and custom post types. When the permalink structure includes `%category%`, the primary category is used for generating post permalinks. Works with any hierarchical taxonomy and supports WP-CLI.

- **Slug**: `easy-primary-category`
- **Text Domain**: `easy-primary-category`
- **Minimum WordPress**: 4.5
- **Minimum PHP**: 5.6
- **Versioning**: [Semantic Versioning](https://semver.org/) (MAJOR.MINOR.PATCH)
- **License**: GPLv2

## Coding Standards

This project follows the **WordPress Coding Standards** enforced via PHPCS.

- Run PHPCS to check coding standards
- Run PHPCBF for auto-fixable violations
- All PHP must comply with the `WordPress` ruleset
- Text domain `easy-primary-category` is enforced by PHPCS

### Naming Conventions

- **Class prefix**: `Easy_Primary_` (e.g., `Easy_Primary_Category`, `Easy_Primary_Term`)
- **Meta key pattern**: `epc_primary_{taxonomy_name}` (e.g., `epc_primary_category`)
- **CSS class prefix**: `easy-` (e.g., `easy-primary-term`, `easy-make-primary-term`)
- **Text domain**: `easy-primary-category` — all translatable strings must use this

### JavaScript

- jQuery-based, follows WordPress JS coding style
- Uses Underscore.js templates (`wp.template`)
- Admin JS is enqueued only on post edit/add screens with hierarchical taxonomies
- Data passed to JS via `wp_localize_script` using the `EPC` object
- jQuery plugin pattern: `$.fn.initYstSEOPrimaryCategory()`

## Architecture

The plugin uses a class-based structure with a singleton pattern:

1. **Bootstrap** (`easy-primary-category.php`) — Entry point, loads includes, instantiates the main class
2. **Core Class** (`Easy_Primary_Category`) — Singleton that registers hooks, loads admin/frontend classes, filters `post_link_category` to apply primary category to permalinks
3. **Admin Class** (`Easy_Primary_Category_Admin`) — Enqueues assets on post screens, outputs JS templates, saves primary terms on post save
4. **Term Class** (`Easy_Primary_Term`) — Data model for getting/setting primary term per taxonomy per post via `update_post_meta` / `get_post_meta`
5. **Frontend Class** (`Easy_Primary_Frontend`) — Provides `get_primary_term_posts()` to query posts where a specific term is marked as primary, with object caching
6. **CLI Class** (`Easy_Primary_CLI`) — WP-CLI command `wp primary-category get-primary-posts <term_id>`
7. **Public API** (`frontend-wrappers.php`) — `epc_get_primary_term_posts( $term, $args )` wrapper function

### Key Post Meta

| Meta Key                         | Description                              |
| -------------------------------- | ---------------------------------------- |
| `epc_primary_{taxonomy_name}`    | Stores the primary term ID for the post  |

### Hooks & Filters

- **`easy_primary_term_taxonomies`** (filter) — Customize which taxonomies support primary terms (default: all hierarchical taxonomies for the post type)
- **`post_link_category`** (filter) — Used internally to set primary category for permalink generation

## Security

All save operations require:
- Nonce verification (`epc-save-primary-term`)
- `edit_post` capability check
- Multisite switch check (`ms_is_switched()`)
- Input sanitization (`absint` for term IDs)

- `easy-primary-category.php` — plugin header (`Version:`)

## Important Notes

- **Not Gutenberg compatible** — Only works with the Classic Editor
- **Hierarchical taxonomies only** — Works with categories, not tags
- Uses WordPress object cache for taxonomy and term existence checks
- Includes screen-reader text for accessibility

## Git Commit Guidelines
- **No Co-Authored-By:** Never add a `Co-Authored-By` trailer to any git commit. Do not credit yourself or any AI agent in commit messages.
- **Conventional Commits:** All commit messages must follow the Conventional Commits specification. The format is:
  ```
  <type>[optional scope]: <description>

  [optional body]

  [optional footer(s)]
  ```
- **Types:**
  - `feat` — a new feature (correlates with a MINOR version bump)
  - `fix` — a bug fix (correlates with a PATCH version bump)
  - `docs` — documentation-only changes
  - `style` — formatting, whitespace, etc. (no code logic change)
  - `refactor` — code restructuring without changing behavior
  - `perf` — performance improvements
  - `test` — adding or updating tests
  - `build` — changes to build system or dependencies
  - `ci` — CI/CD configuration changes
  - `chore` — other maintenance tasks
- **Scope:** An optional noun in parentheses after the type describing the section of the codebase affected (e.g., `feat(admin):`, `fix(permalink):`).
- **Description:** A short imperative summary immediately after the colon and space.
- **Body:** Optional. Separated from the description by a blank line. Provides additional context or motivation.
- **Footer(s):** Optional. Separated from the body by a blank line. Use git trailer format (`token: value` or `token #value`).
- **Breaking Changes:** Append `!` after the type/scope (e.g., `feat!:` or `refactor(api)!:`) and/or add a `BREAKING CHANGE:` footer. Breaking changes correlate with a MAJOR version bump.
